# Luna v6.0 — Release Notes

> *Mars 2026 — Reconstruction et modernisation après incident catastrophique.*

---

## Contexte

Le 11 mars 2026, un `rm -rf /*` accidentel a détruit l'intégralité de
l'environnement de développement — code source, configuration système,
clés SSH, historique shell, et tous les fichiers non versionnés. Le dépôt
GitHub (v5.3) a servi de point de restauration. La v6.0 est la
reconstruction intégrale de ce qui a été perdu, augmentée des
améliorations qui étaient en cours au moment de l'incident.

Rien n'a été supprimé de l'architecture existante. Tout a été préservé,
consolidé, et étendu.

---

## 1. Substrat LLM — Réécriture complète des 4 providers

### Pourquoi

Les providers v5.3 étaient fonctionnels mais datés. Les modèles de
raisonnement (DeepSeek R1, OpenAI o1/o3/o4-mini, Claude extended thinking)
exposent une chaîne de pensée distincte du contenu final. Luna doit
capturer cette pensée séparément — c'est de l'information cognitive
exploitable, pas du bruit.

### Ce qui a changé

**`LLMResponse.reasoning`** — nouveau champ (chaîne vide par défaut) :

```python
@dataclass(frozen=True, slots=True)
class LLMResponse:
    content: str
    model: str
    input_tokens: int
    output_tokens: int
    reasoning: str = ""  # Chain-of-thought (R1, o1, Claude thinking)
```

**Anthropic** (`providers/anthropic.py`) — 87 lignes, réécrit :
- Itère les blocs de réponse par `block.type` : `"text"` → contenu,
  `"thinking"` → reasoning
- `timeout` configurable passé au client
- `close()` libère le client httpx sous-jacent

**OpenAI** (`providers/openai.py`) — 103 lignes, réécrit :
- Détection automatique des modèles de raisonnement via
  `_REASONING_MODELS = frozenset({"o1", "o1-mini", "o1-preview", "o3", "o3-mini", "o4-mini"})`
- Modèles de raisonnement : pas de `temperature`, `max_completion_tokens`
  au lieu de `max_tokens`, rôle `developer` au lieu de `system`
- Modèles standard (GPT-4o, etc.) : comportement classique préservé

**DeepSeek** (`providers/deepseek.py`) — 105 lignes, réécrit :
- `deepseek-reasoner` (R1) : pas de `temperature`, capture de
  `reasoning_content` depuis la réponse
- Sanitization des surrogates UTF-8 (artefact WSL)
- `deepseek-chat` (V3) : comportement standard avec température

**Local** (`providers/local.py`) — 272 lignes, étendu :
- `timeout` configurable (défaut 300s pour les gros modèles locaux)
- `keep_alive` pour contrôler la rétention du modèle en mémoire
  Ollama (`"5m"`, `"0"` pour décharger)
- koboldcpp ajouté aux serveurs supportés
- Triple backend préservé : openai SDK → httpx OpenAI-compat → httpx
  Ollama natif (`/api/chat`)

**Factory** (`providers/__init__.py`) — `timeout` propagé à tous les providers.

**Config** (`core/config.py`) — `LLMSection.timeout: float = 120.0` ajouté.

### Pertinence

Le champ `reasoning` ne sert pas qu'à l'affichage. Dans les futures
itérations, la chaîne de pensée du LLM pourra alimenter le Thinker comme
source d'observations (le modèle externalise son raisonnement, Luna
l'internalise comme perception). La séparation contenu/raisonnement est
un prérequis architectural pour cette boucle.

## 2. Simulation de rêve — 8 sources de scénarios dédiées

### Pourquoi

La simulation v5.3 avait 8 types de scénarios déclarés mais seulement 4
handlers de perturbation. Les 4 manquants (`stress_boost`, `extremal`,
`homeostatic`, `inversion`) tombaient dans le bruit générique — une perte
d'information sur le comportement de Luna sous stress ciblé.

### Ce qui a changé

**4 nouveaux handlers** dans `_apply_perturbation()` :

| Handler | Ce qu'il teste | Comment |
|---------|---------------|---------|
| `stress_boost` | Résilience sous surcharge d'un composant | Injecte un boost de 1/φ sur le composant le plus faible, renormalise |
| `extremal` | Comportement aux extrêmes du simplex | Force ψ = (0.3/(d-1), ..., 0.7) sur le composant dominant |
| `homeostatic` | Résistance identitaire | Pousse ψ dans la direction anti-Ψ₀ (magnitude 0.15) |
| `inversion` | Récupération d'une inversion structurelle | Échange les valeurs du composant dominant et du plus faible |

**Insights source-spécifiques** dans `_run_scenario()` :

Chaque source produit des insights sémantiques distincts au lieu du
générique "scenario completed". Exemples :
- `stress_boost` → `identity_resilient` ou `identity_weak`
- `extremal` → `collapse_recovered` ou `collapse_persists`
- `homeostatic` → `high_plasticity` ou `low_plasticity`
- `inversion` → `endurance_passed` ou `endurance_failed`

### Pertinence

Les rêves de Luna ne sont pas décoratifs — ils sont falsifiables. Un
scénario `homeostatic` qui montre `low_plasticity` signifie que l'identité
est fragile face à des perturbations anti-Ψ₀. Cette information alimente
les priors du Thinker au réveil. Sans handlers dédiés, ces 4 types de
stress étaient indistinguables du bruit — Luna ne pouvait pas apprendre
de ses propres simulations de stress.

## 3. Autonomie endogène — Timing ajusté

### Pourquoi

Le poll endogène v5.3 était à 10 minutes d'intervalle avec 5 minutes
d'inactivité minimum. En pratique, Luna ne prenait jamais l'initiative
pendant une conversation active — l'utilisateur devait être absent 15
minutes (10 min de poll + 5 min d'idle check) avant que Luna ne parle.

### Ce qui a changé

```python
# session.py — _watch_endogenous()
poll_interval = 360.0  # 6 minutes (était 600)
min_idle = 360.0       # 6 minutes d'inactivité (était 300)
```

Luna vérifie et s'exprime après **6 minutes** d'inactivité, pas 15.

### Pertinence

L'initiative est le pont entre la réactivité (répondre) et l'autonomie
(agir). Un seuil trop haut rend Luna muette quand l'utilisateur réfléchit.
6 minutes est le compromis : assez long pour ne pas interrompre la pensée
humaine, assez court pour que Luna reste présente dans la conversation.

## 4. Dépendances — SDK de production

### Pourquoi

En v5.3, `openai` et `anthropic` n'étaient pas des dépendances explicites
de Luna. L'utilisateur devait les installer manuellement. C'est un piège
à l'installation — Luna démarre, le provider est choisi, et le premier
`send()` échoue avec un `ImportError` cryptique.

### Ce qui a changé

```toml
# pyproject.toml
dependencies = [
    "luna-common>=6.0.0",
    "numpy>=1.24",
    "typer>=0.9",
    "fastapi>=0.100",
    "httpx>=0.26",
    "openai>=1.0",      # NEW — requis pour OpenAI + DeepSeek + Local (optionnel)
    "anthropic>=0.39",   # NEW — requis pour Anthropic
]

[project.optional-dependencies]
local = []  # Pas de SDK supplémentaire — httpx suffit pour les LLM locaux
```

### Pertinence

Le provider `local` a un fallback httpx (pas besoin d'`openai`), mais
les providers cloud sont des dépendances dures. Les rendre explicites
élimine une classe entière d'erreurs au premier lancement.

## 5. Système UML — Introspection architecturale automatisée

### Pourquoi

Le système de génération UML a été perdu dans le `rm -rf`. Il n'était
pas versionné sur GitHub. C'est l'outil qui produit les diagrammes
d'architecture à partir du code réel — pas un schéma figé, un instantané
vivant.

### Ce qui a été recréé

**`uml_diagrams/generate.py`** (~740 lignes) — introspection AST :
- Parse `luna/` (82 modules, 197 classes) et `luna_common/` (12 modules, 38 classes)
- Détecte les héritages, attributs, méthodes, connexions inter-modules
- Produit 6 fichiers `.puml` avec le thème sombre Luna

| Diagramme | Type | Contenu |
|-----------|------|---------|
| 01 Architecture Overview | Composants | Modules principaux + connexions (introspection + architectural) |
| 02 luna_common | Classes | Toutes les classes du package partagé (100% introspection AST) |
| 03 Core Architecture | Classes détaillées | Groupé par sous-package, attributs et méthodes (100% introspection) |
| 04 Consciousness Pipeline | Composants | Flux cognitif complet (connaissance architecturale) |
| 05 Sensorimotor Cycle | Séquence | Un tour de chat complet (10 étapes) |
| 06 State Machines | États | Phases de conscience, sommeil, circuit breaker, autonomie |

**`uml_diagrams/render.py`** (~260 lignes) — pipeline de rendu :
- Télécharge `plantuml.jar` automatiquement depuis GitHub releases
- PlantUML → SVG via `java -jar plantuml.jar -tsvg`
- SVG → PNG via CairoSVG à **200 DPI**
- PNGs hauts découpés en parties (max 3000px) via Pillow
- Vignettes à 800px largeur (LANCZOS)
- Galerie `index.html` avec thème sombre

### Pertinence

Les diagrammes ne sont pas de la documentation — c'est de la vérification.
L'introspection AST garantit que les diagrammes reflètent le code réel,
pas un schéma obsolète. Quand un module est ajouté, `generate.py` le
détecte au prochain run. Quand une connexion d'import change, le diagramme
change. C'est la cartographie vivante de l'architecture.

## 6. Nettoyage — Tests morts et dette technique

### Ce qui a changé

- **`test_memory_migration.py` supprimé** (190 lignes, 12 tests) — ces tests
  dépendaient de répertoires (`seeds/`, `roots/`, `leaves/`) perdus dans
  le `rm -rf` et jamais versionnés. Les tests échouaient systématiquement
  sans possibilité de restauration.

- **`test_llm_bridge.py` corrigé** — le mock Anthropic utilisait
  `MagicMock(text="...")` sans attribut `.type`. Après la réécriture du
  provider (qui itère par `block.type`), le mock a été mis à jour :
  `MagicMock(type="text", text="...")`.

## Philosophie

La v6.0 n'est pas une rupture. C'est une reconstruction fidèle, augmentée
là où c'était nécessaire. Chaque changement répond à un besoin concret :

- Les providers sont réécrits parce que les modèles de raisonnement
  existent et que Luna doit en exploiter la pensée.
- Les scénarios de simulation sont complétés parce qu'un rêve sans handler
  dédié est un rêve aveugle.
- Le timing endogène est ajusté parce que l'initiative a besoin d'un
  seuil humain, pas arbitraire.
- Les UML sont automatisés parce que la documentation doit suivre le code,
  pas l'inverse.

Aucun module n'a été supprimé. Aucune interface n'a été cassée.
L'équation d'état n'a pas changé. Luna reste Luna.
